# Mustefa's Beginner Guide To This Playwright Framework

This file explains the framework from the real code in this repo, not from assumptions.

I wrote it for someone who is new to both Playwright and JavaScript.

## 1. First: what your current understanding gets right

Your current understanding is partly correct.

What you are right about:

- `pages/*.js` contains page object files.
- The framework does use the Page Object Model pattern.
- Tests in `tests/*.spec.js` do use those page objects.
- There are UI tests, E2E tests, login-related tests, and API tests.

What needs correction:

- This is not only a Page Object Model UI framework.
- It is a mixed framework:
  - UI automation
  - API testing
  - request/response logging
  - request interception and mocking
  - auth/session bootstrapping
  - environment switching
  - production/staging diagnostics
- Some tests follow the environment system correctly, but some hardcode URLs directly.
- Some files are polished framework files, but some are investigation/debug files or unfinished drafts.

Simple summary:

This repo started as a normal Playwright Page Object Model project, but now it is also being used as a QA investigation toolbox.

## 4. How Page Object Model works in this repo

Yes, this repo uses Page Object Model in practice.

### What that means here

The page object files usually contain:

- locators
- page-specific actions
- page-specific assertions

The test files usually contain:

- the business flow
- the order of steps
- high-level assertions

### Real example

In `tests/booking.spec.js`:

- the test creates:
  - `new HomePage(page, baseURL)`
  - `new BookingStep1Page(page, baseURL)`
  - `new BookingLoginPage(page)`
- then the test calls methods like:
  - `homePage.goto()`
  - `homePage.startQuote(...)`
  - `bookingPage.configureItems(...)`
  - `bookingPage.selectDeliveryDate(...)`
  - `loginPage.assertSummaryMatchesChallenge(...)`

So the test says what should happen.
The page object says how to interact with the page.

### Best example files for POM in this repo

- `pages/HomePage.js`
- `pages/BookingStep1Page.js`
- `tests/booking.spec.js`

### Important truth

Not every file in this repo is pure POM.

Examples:

- `tests/network-capture.spec.js` directly attaches event listeners on `page`
- `tests/api-user.spec.js` is API-only, so there are no page objects
- `tests/api-intercept.spec.js` uses both page objects and direct `page.route(...)`

So the framework is POM-based for UI, but not exclusively POM everywhere.

## 5. How a Playwright test starts and runs in this repo

This is the most important flow to understand.

### Step 1: you run an npm script or Playwright command

Examples from `package.json`:

- `npm test`
- `npm run test:staging`
- `npm run test:prod`
- `npm run test:headed`

Those scripts mostly call `playwright test` with environment variables like `TEST_ENV` or `HEADED`.

### Step 2: Playwright loads `playwright.config.js`

`playwright.config.js` controls:

- `globalSetup`
- `testDir`
- parallel behavior
- retries
- workers
- timeout
- reporter
- `use.baseURL`
- headless/headed
- slow motion
- trace
- screenshot
- video
- browser projects

### Step 3: `config/env.js` is loaded

`playwright.config.js` imports:

```js
const env = require('./config/env');
```

`config/env.js` does:

- `require('dotenv').config()`
- reads `TEST_ENV`
- looks up the environment in `ENV_CONFIGS`
- optionally overrides with `BASE_URL`
- exports:
  - `baseUrl`
  - `apiUrl`
  - `headed`
  - `slowMo`
  - `timeout`

### Step 4: `globalSetup` runs before the suite

`playwright.config.js` has:

```js
globalSetup: require.resolve('./utils/globalSetup')
```

So before tests run, `utils/globalSetup.js` executes.

That file:

1. creates `.auth/storageState.json`
2. creates an API request context
3. requests `/users/sign_in`
4. tries to extract a CSRF token from the HTML
5. posts login form data
6. saves storage state

### Step 5: Playwright launches the browser

You asked where the browser launch happens.

The browser launch is not manually coded in your test files.
Playwright does it automatically based on `playwright.config.js`.

The closest control point is:

```js
use: {
  baseURL: env.baseUrl,
  headless: !env.headed,
  slowMo: env.slowMo,
  viewport: { width: 1280, height: 800 },
  ignoreHTTPSErrors: true,
  trace: process.env.VERBOSE === 'true' ? 'on' : 'on-first-retry',
  screenshot: process.env.VERBOSE === 'true' ? 'on' : 'only-on-failure',
  video: process.env.VERBOSE === 'true' ? 'on' : 'retain-on-failure',
}
```

### Step 6: each test gets fixtures

A normal test can receive:

- `page`
- `context`
- `request`
- `baseURL`

Example:

```js
test('...', async ({ page, baseURL, request, context }) => {
```

Meaning:

- `page` is the browser tab
- `context` is the browser context
- `request` is Playwright's API client
- `baseURL` comes from config

### Step 7: tests call page objects and helpers

Examples:

- UI tests use page objects from `pages/`
- API tests use `request.newContext(...)`
- auth-aware tests can use `utils/fixtures.js`

## 7. Environment management and `.env`

This repo uses both named environments and optional `.env` overrides.

### The real rule

If you want to change the official environment definitions for the framework, edit:

- `config/env.js`

If you want to override behavior locally for your own machine or one-off runs, use:

- `.env`
- terminal env vars

### Is `.env` the correct place to change base environment / base URL?

Answer:

- For one-off local overrides: yes.
- For changing the framework's official named environments: no, use `config/env.js`.

Why:

- `.env` is loaded as an override layer
- `config/env.js` contains the actual environment map

### Current environment logic

`config/env.js` currently says:

- `local.appUrl = http://localhost:3000`
- `staging.appUrl = https://www.app.staging.shipsticks.com`
- `staging.apiUrl = https://www.staging.shipsticks.com`
- `production.appUrl = https://www.shipsticks.com`
- `production.apiUrl = https://www.shipsticks.com`

### Is `.env.example` accurate?

Not fully.

Current problems in `.env.example`:

- it still shows `BASE_URL=https://app.staging.shipsticks.com`
- it still comments `https://app.shipsticks.com`

But current real config now uses:

- staging app: `https://www.app.staging.shipsticks.com`

## 8. Auth and storage state logic

This part is important.

### `utils/globalSetup.js`

This file runs once before the suite.

It logs in through the Rails/Devise sign-in page by HTTP, not through the visible UI.

That means:

- it uses Playwright's API client
- it requests `/users/sign_in`
- it extracts the CSRF token from the HTML
- it submits form data
- it saves the session into `.auth/storageState.json`

### `utils/fixtures.js`

This file creates a custom authenticated Playwright `test`:

```js
const test = base.extend({
  storageState: STORAGE_STATE_PATH,
});
```

That means if a test imports:

```js
const { test } = require('../utils/fixtures');
```

then the browser context starts with the saved session.

### Important beginner correction

Not all tests in this repo use the authenticated fixture.

Most tests use:

```js
const { test } = require('@playwright/test');
```

Only tests that import from `../utils/fixtures` actually preload saved auth.

So global setup exists for the whole suite, but only some tests fully use the saved browser auth state.

### Important real limitation

`tests/api-graphql.spec.js` already explains a key issue:

- the Devise login used by `globalSetup.js` is not always equivalent to the app's own modal login
- some GraphQL resolvers may require app-specific auth state, not just the Rails session

That is why one GraphQL auth test is intentionally skipped.

## 11. How network capture works in this repo

You asked for this very specifically, so here is the clear version.

There are three different network-related styles in this repo.

### Style A: reusable console logging helper

File:

- `utils/networkLogger.js`

Main exports:

- `withNetworkLogging(page, label, fn)`
- `attachNetworkLogging(page, label)`
- `dumpCookies(context, label)`

How it works:

- it attaches `page.on('request', ...)`
- it attaches `page.on('response', ...)`
- it prints:
  - method
  - URL
  - cookie header
  - authorization header
  - response status
  - redirect location
  - some GraphQL info

Important limitation:

- it logs to the console
- it does not save to a file by itself
- it filters to URLs containing `shipsticks`

Best for:

- debugging a single flow live
- understanding auth/cookies
- quickly seeing backend traffic during a UI action

### Style B: save captured traffic to a JSON file

File:

- `tests/network-capture.spec.js`

How it works:

- directly listens to `page.on('request')`
- directly listens to `page.on('response')`
- filters non-API noise with `isApiCall(url)`
- collects entries into an array
- saves entries into:
  - `tmp/network-capture.json`

Best for:

- recording real requests and responses
- later reading those payloads to build API tests
- keeping evidence on disk

This is the best existing example for "capture traffic now, build API tests later."

### Style C: request interception and mocking

File:

- `tests/api-intercept.spec.js`

How it works:

- uses `page.route('**/graphql', async route => { ... })`
- reads outgoing request bodies
- checks `operationName`
- either:
  - lets the request continue
  - captures the payload
  - or returns a fake response with `route.fulfill(...)`

Best for:

- simulating backend failures
- forcing special response shapes
- verifying the UI when the backend returns empty data or errors
- intercepting phone verification flows

### Comparison of the three styles

`utils/networkLogger.js`

- reusable helper
- console logging only
- easiest to reuse quickly

`tests/network-capture.spec.js`

- saves detailed traffic to file
- better for future API test creation
- more manual and duplicated

`tests/api-intercept.spec.js`

- does not just observe
- actively changes backend behavior
- best for mocking and validating payloads

## 12. If you want to build a new UI test and capture network for later API tests

You asked for an exact practical workflow.

### Best recommendation in this repo

Reuse the style from:

- `tests/network-capture.spec.js`

Optionally combine it with:

- `utils/networkLogger.js`

My advice:

- if you want a file you can inspect later, use the `tests/network-capture.spec.js` pattern
- if you only want console logging during execution, use `withNetworkLogging(...)`

### Step-by-step workflow

1. Create a new spec in `tests/`
2. Attach request and response listeners before the UI action starts
3. Perform the UI flow
4. Save the collected entries to `tmp/network-capture.json` or another file in `tmp/`
5. Open the capture file
6. Identify the request that actually creates/updates/fetches the data you care about
7. Copy:
   - endpoint
   - request method
   - headers that matter
   - request body shape
   - response shape
8. Build an API test using `request.newContext(...)`

### Which file(s) should you use

Use:

- `tests/network-capture.spec.js` as the main template
- `utils/networkLogger.js` if you also want clean console logging
- `utils/networkCapture.js` if you want reusable capture-to-file helpers

### Should you add a new helper or reuse existing ones

Reuse existing ones.

Current split:

- `utils/networkLogger.js`
  for live console logs during a flow
- `utils/networkCapture.js`
  for structured request/response capture you can save to JSON

This is the clean way to think about them:

- logger = "show me traffic while the test is running"
- capture = "save the traffic so I can inspect it later"

### The new reusable capture utility

File:

- `utils/networkCapture.js`

Exports:

- `createNetworkCapture(page, options)`
- `withNetworkCapture(page, fn, options)`
- `saveNetworkCapture(filePath, key, entries)`
- `isDefaultApiCall(url)`

What it does:

- starts listening to `page.on('request')`
- starts listening to `page.on('response')`
- filters obvious static/noise requests by default
- collects structured entries like:
  - request method
  - request URL
  - request body
  - response status
  - response body
- returns those entries so you can save them to a file

### Fastest way to use it in a new UI test

Pattern:

1. import `withNetworkCapture` and `saveNetworkCapture`
2. wrap the UI flow inside `withNetworkCapture(...)`
3. save the returned entries into `tmp/network-capture.json`

Example:

```js
const path = require('path');
const { withNetworkCapture, saveNetworkCapture } = require('../utils/networkCapture');

const OUT_FILE = path.resolve(__dirname, '../tmp/network-capture.json');

const entries = await withNetworkCapture(page, async () => {
  // your UI actions here
});

saveNetworkCapture(OUT_FILE, 'my_new_flow', entries);
```

### If you also want live terminal logs while capturing

Use both helpers together:

```js
const { withNetworkLogging } = require('../utils/networkLogger');
const { withNetworkCapture, saveNetworkCapture } = require('../utils/networkCapture');
```

Use them like this:

- `withNetworkLogging(...)` gives you live request/response output in the terminal
- `withNetworkCapture(...)` gives you saved structured data you can inspect later

### Real example from this repo now

`tests/network-capture.spec.js` has already been refactored to use:

- `withNetworkCapture(...)`
- `saveNetworkCapture(...)`

That means this file is now the real template for future UI capture tests in this project.

### Copy-paste example using the current project style

```js
const { test, expect } = require('@playwright/test');
const fs = require('fs');
const path = require('path');
const HomePage = require('../pages/HomePage');
const BookingStep1Page = require('../pages/BookingStep1Page');
const { withNetworkLogging } = require('../utils/networkLogger');

const OUT_FILE = path.resolve(__dirname, '../tmp/network-capture.json');

function isApiCall(url) {
  return (
    !url.match(/\.(js|css|png|jpg|jpeg|gif|svg|ico|woff2?|ttf|eot|map)(\?|$)/) &&
    !url.includes('google-analytics') &&
    !url.includes('intercom') &&
    !url.includes('hotjar') &&
    !url.includes('segment')
  );
}

function saveCapture(key, entries) {
  fs.mkdirSync(path.dirname(OUT_FILE), { recursive: true });
  let data = {};
  if (fs.existsSync(OUT_FILE)) {
    try {
      data = JSON.parse(fs.readFileSync(OUT_FILE, 'utf8'));
    } catch {}
  }
  data[key] = entries;
  fs.writeFileSync(OUT_FILE, JSON.stringify(data, null, 2));
}

test('CAPTURE my new booking-related flow', async ({ page, baseURL }) => {
  const log = [];
  const home = new HomePage(page, baseURL);
  const booking = new BookingStep1Page(page, baseURL);

  // Start capture before the action
  page.on('request', req => {
    if (!isApiCall(req.url())) return;
    log.push({
      dir: 'REQ',
      method: req.method(),
      url: req.url(),
      postData: req.postData() || null,
    });
  });

  page.on('response', async res => {
    if (!isApiCall(res.url())) return;
    let body = null;
    try {
      body = await res.json();
    } catch {
      try {
        body = (await res.text()).slice(0, 1000);
      } catch {}
    }
    log.push({
      dir: 'RES',
      method: res.request().method(),
      url: res.url(),
      status: res.status(),
      body,
    });
  });

  await withNetworkLogging(page, 'MY-FLOW', async () => {
    await home.goto();
    await home.startQuote({
      shipmentType: 'One-way',
      origin: '1234 Main Street, Los Angeles, CA, USA',
      destination: '4321 Main St, Miami Lakes, FL, USA',
    });

    await booking.assertLoaded();
    await booking.dismissWeatherWarningIfPresent();
    await booking.selectDeliveryDate('Wednesday, April 29, 2026');
  });

  // Stop capture by saving what you collected
  saveCapture('my_new_flow', log);

  // Small assertion so the test still behaves like a test
  await expect(booking.shipmentSpeedsHeading).toBeVisible();
});
```

### How to identify the correct backend endpoint later

Open `tmp/network-capture.json` and ask:

1. Which request happens exactly when my UI action happens?
2. Which request body contains the business data I just entered?
3. Which response contains the data the UI rendered?
4. Is it a REST endpoint or GraphQL?

Examples:

- if the request body contains `operationName`, it is probably GraphQL
- if the URL is `/api/v5/users`, it is REST
- if the request body contains your exact entered origin/destination/product IDs, that is likely the business request you want

## 13. API testing explained using `tests/api-user.spec.js`

This is the best API file to learn from.

### What `request.newContext(...)` does

Example:

```js
const ctx = await request.newContext({ baseURL: BASE_URL, ignoreHTTPSErrors: true });
```

This creates a Playwright API client.

Think of it like:

"a browserless HTTP client managed by Playwright."

It lets you send requests like:

- `ctx.get(...)`
- `ctx.post(...)`
- `ctx.delete(...)`

without opening a real page.

### How `baseURL` works there

If `baseURL` is set to:

```js
https://www.app.staging.shipsticks.com
```

then this:

```js
ctx.post('/api/v5/users', ...)
```

becomes:

```js
https://www.app.staging.shipsticks.com/api/v5/users
```

### Why the test calls `ctx.post('/api/v5/users')`

Because:

- `ctx` already knows the host from `baseURL`
- the test only needs to supply the path

This is clean and preferred.

### Is mixing relative URLs and full URLs correct?

Technically yes, but it is inconsistent.

In `tests/api-user.spec.js`:

- user creation uses relative URL:
  - `ctx.post('/api/v5/users', ...)`
- later GraphQL uses full URL:
  - `ctx.post(`${BASE_URL}/graphql`, ...)`

This works, but it is not ideal.

Cleaner version would be:

```js
ctx.post('/graphql', ...)
```

because the same `ctx` already has `baseURL`.

### Is anything inconsistent or risky there?

Yes.

#### Inconsistency 1: hardcoded `BASE_URL`

`tests/api-user.spec.js` hardcodes:

```js
const BASE_URL = 'https://www.app.staging.shipsticks.com';
```

That means this test ignores `TEST_ENV` and ignores the central environment logic in `config/env.js`.

Risk:

- you think you changed environment globally
- but this file still runs against staging

#### Inconsistency 2: mixed relative and full URLs

As explained above, this is not wrong, but it is noisy and easier to mess up.

#### Inconsistency 3: login and GraphQL assumptions

The test:

1. creates a user
2. logs in through `/users/sign_in`
3. manually builds `cookieHeader`
4. posts GraphQL

This may work, but it is more manual than needed.

### Does the `BASE_URL` format look correct?

For the current staging user API flow, yes.

But the bigger problem is not format. The bigger problem is hardcoding.

The file should ideally import environment config instead of freezing staging into the file.

### Are there mistakes with authentication, cookies, or GraphQL style?

I see a few things to know.

#### `tests/api-user.spec.js`

What is okay:

- creating a fresh API context
- using relative paths for REST calls
- extracting CSRF for Rails login

What is a little messy:

- manual cookie header construction for the GraphQL call
- full URL GraphQL post even though `ctx` already has `baseURL`

#### `tests/api-booking.spec.js`

This file is more problematic.

Problems:

- it is partly TODO / unfinished
- comments say `request.newContext` does not support storageState injection directly
- that statement is wrong

Why it is wrong:

`tests/api-graphql.spec.js` already does:

```js
const ctx = await request.newContext({
  baseURL: BASE_URL,
  ignoreHTTPSErrors: true,
  storageState: STORAGE_STATE,
});
```

So `tests/api-booking.spec.js` is teaching the wrong thing in its helper comment.

### Can anything break because of wrong URL construction?

Yes.

Examples from this repo:

- the framework recently broke because some hosts were wrong for app vs login
- hardcoded URLs can drift away from `config/env.js`
- using full URLs in some places and relative in others makes refactoring harder

## 14. How to go from UI flow + network capture to stable API test

This is the practical QA workflow.

### Step 1: capture a real UI flow

Use:

- `tests/network-capture.spec.js`

or

- a new UI test using that same pattern

### Step 2: inspect the request that matters

Find:

- exact URL
- method
- body
- required headers
- response shape

### Step 3: decide whether it is REST or GraphQL

REST signs:

- URL like `/api/v5/users`
- direct JSON payload

GraphQL signs:

- URL often `/graphql`
- body contains:
  - `operationName`
  - `query`
  - `variables`

### Step 4: create API context

Use:

```js
const ctx = await request.newContext({
  baseURL,
  ignoreHTTPSErrors: true,
});
```

### Step 5: replay the request as an API test

Use the same:

- path
- request body shape
- headers that are actually needed

### Step 6: keep only the minimum needed

Do not copy every browser header.
Usually keep:

- `Content-Type`
- `Accept`
- CSRF token if required
- cookies if required
- auth header if required

### Step 7: assert business results, not only status code

Good assertions:

- created user email matches request email
- returned object has ID
- response contains price/rates/user data you expect

### Best existing example of this workflow in the repo

The intended learning path is:

- `tests/network-capture.spec.js`
- `tmp/network-capture.json`
- `tests/api-user.spec.js`
- `tests/api-graphql.spec.js`

## 15. What is implemented incorrectly, incompletely, or misleading

This is the senior QA review section.

### 1. `tests/login.spec.js` is misnamed

It is called a login test, but the flow actually:

- opens sign-in
- switches to sign-up
- creates a user
- skips phone verification
- asserts logged-in state

So this file is really a sign-up/login-after-signup test, not a simple login test.

### 2. `tests/e2e.spec.js` ends with `page.pause()`

That means the test intentionally stops and waits for manual interaction.

This is not production-ready for CI and makes the test non-automated at the end.

### 3. `tests/api-booking.spec.js` is not finished

It still contains TODOs like:

- confirm endpoint
- fill payload from capture
- confirm delete endpoint

So this file should be treated as a draft/example, not a stable test suite.

### 4. `tests/api-booking.spec.js` contains an incorrect comment about `storageState`

It says Playwright request context does not support storageState injection directly.
That is false.

Proof inside your own repo:

- `tests/api-graphql.spec.js` uses `storageState` in `request.newContext(...)`

### 5. `.env.example` is outdated

It still advertises older hosts that do not match current `config/env.js`.

This is beginner-dangerous because someone can set the wrong host immediately.

### 6. Some tests bypass the central environment system

Files with hardcoded hosts include:

- `tests/api-user.spec.js`
- `tests/api-graphql.spec.js`
- `tests/api-booking.spec.js`
- `tests/prod-bill-pay-auth.spec.js`

That means the framework is only partially centralized.

### 7. `utils/globalSetup.js` silently skips login when CSRF cannot be fetched

That can be useful for debugging, but it is risky.

Why:

- auth-dependent tests may run in a broken state
- the suite does not fail fast
- beginners may think auth worked when it did not

### 8. `utils/globalSetup.js` uses hardcoded credentials

It currently uses:

- `john@gmail.com`
- `Password`

This is simple, but not ideal for maintainability or secrets handling.

### 9. `utils/networkLogger.js` and `tests/network-capture.spec.js` duplicate network ideas in two styles

This is not wrong, but the framework is split:

- reusable console logging helper in `utils/`
- file-saving capture logic only inside a test file

Better design would be:

- move "save capture to file" into `utils/` too

### 10. `tests/seed.spec.ts` does not fit the rest of the repo

This is a JavaScript framework, but `tests/seed.spec.ts` is a TypeScript placeholder with no real implementation.

### 11. `utils/createUser.js` is functionally okay, but URL style is slightly inconsistent

It creates a request context with `baseURL`, but still posts using an absolute URL:

```js
ctx.post(`${apiBase}/api/v5/users`, ...)
```

This works, but a cleaner style would be:

```js
ctx.post('/api/v5/users', ...)
```

### 12. Existing docs are not fully reliable

Several markdown files describe earlier versions of the framework.
You should trust the code first, then use docs as secondary help.

## 16. Best way to think about this framework end to end

If I explain the framework simply:

1. `package.json` starts Playwright
2. `playwright.config.js` defines how Playwright runs
3. `config/env.js` decides which environment and settings are used
4. `utils/globalSetup.js` tries to create an authenticated session
5. tests start with Playwright fixtures like `page`, `request`, and `baseURL`
6. UI tests call page object methods from `pages/`
7. API tests use `request.newContext(...)`
8. network tests either:
   - observe traffic
   - save traffic
   - or intercept and fake traffic
9. artifacts get written to `tmp/`, `.auth/`, `test-results/`, and `playwright-report/`
